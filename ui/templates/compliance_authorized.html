✅ How Scope Enforcement Works

// Zero Trust by Default

No target is scannable until explicitly added to the Targets registry as "Authorized". This applies to every job creation path — individual scans, the 7 sequential Workflows, and all 3 Pivot Chains.

// Supported Scope Types

IP address, CIDR range, exact domain, or wildcard domain (subdomains). Unauthorized entries always override authorized ones on conflict (fail-safe), and the most specific rule wins.

// Who Can Modify Scope

Only the admin role can add or remove targets — enforced server-side, not just hidden in the UI. This reflects real-world engagement practice: scope is defined once by the lead/admin per the signed Rules of Engagement, not improvised by individual analysts.

// Real-World Nuance

On a real engagement, scope is formally defined in a signed contract (Rules of Engagement) before any testing begins — this toolbox's Targets registry is the technical enforcement of that legal agreement, not a replacement for it.
